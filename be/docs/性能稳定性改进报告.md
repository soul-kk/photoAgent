## 三、各项改进说明

### 3.1 Kimi 并发信号量（`kimigate`）

**问题**：无并发上限时，多用户同时分析会并行占用内存（大图 Base64）和 Kimi 长连接，易触发上游 429 或 OOM。

**实现**：
- `kimigate.Gate` 使用带缓冲 channel 限制同时进行的 Kimi 请求数（默认 5）。
- 排队超过 `queue_wait_sec`（默认 30s）返回 `503`，响应头带 `Retry-After: 30`。
- 所有 `postKimiChat` / `postKimiChatStream` 入口统一 Acquire/Release。

**配置**（`config/app.yaml`）：
```yaml
kimi:
  max_concurrent: 5
  queue_wait_sec: 30
```

---

### 3.2 HTTP 连接池调优

**问题**：原 `http.Client{Timeout: 300s}` 使用默认 Transport，高并发时连接复用不足。

**实现**（`kimigate/gate.go`）：
- `MaxIdleConns: 100`
- `MaxIdleConnsPerHost: 20`
- `IdleConnTimeout: 90s`
- `KeepAlive` 拨号、TLS 握手超时

---

### 3.3 超时与压缩配置外置

**问题**：超时 300s、长边 1024、JPEG 80 写死在代码中，调优需改代码重编译。

**实现**：

| 配置键 | 默认值 | 作用 |
|--------|--------|------|
| `kimi.timeout_sec` | 300 | Kimi HTTP 总超时 |
| `kimi.max_concurrent` | 5 | 最大并发 Kimi 请求 |
| `kimi.queue_wait_sec` | 30 | 等槽位超时 |
| `kimi.score_max_tokens` | 800 | 评分接口输出上限 |
| `image.max_long_edge` | 1024 | 上传前压缩长边 |
| `image.jpeg_quality` | 80 | JPEG 质量 |
| `app.shutdown_sec` | 60 | 优雅停机等待 |

未配置或 ≤0 时使用上表默认值。

---

### 3.4 score-image 限制 max_tokens

**问题**：评分接口未限制 `max_tokens`，模型可能输出过长 JSON，延迟不可控。

**实现**：`PhotographyScoreImage` 使用 `kimi.score_max_tokens`（默认 800），与提示词「text_analysis 200–500 字」对齐。

---

### 3.5 流式：客户端断开取消上游

**问题**：原流式使用 `WithoutCancel`，用户关闭页面后 Kimi 仍跑满 300s，浪费配额与 goroutine。

**实现**：
- **非流式**：仍用 `UpstreamContext`（`WithoutCancel` + 超时），避免网关先断导致半失败。
- **流式**：用 `StreamUpstreamContext`（绑定客户端 `Request.Context`）；扫描 SSE 时检测 `upCtx.Done()` 并退出。

---

### 3.6 压缩后释放原图内存

**问题**：20MB 原图与压缩结果、Base64 可能同时驻留内存。

**实现**：`imageBinaryToDataURL` 成功后，各 handler 执行 `raw = nil`，便于 GC 尽早回收。

涉及：`RecognizeImageBinary`、`PhotographyAnalyzeImage`、`PhotographyScoreImage`、`PhotographyShootAdvice`。

---

### 3.7 优雅停机

**问题**：原 `Shutdown` 仅等 5s，而 Kimi 请求最长 300s，部署时会硬断进行中的分析。

**实现**：`app.shutdown_sec` 默认 60s（可改为 120+ 若生产环境需要）。

---

## 四、行为变化（对调用方）

| 场景 | 变化 |
|------|------|
| 并发超过 `max_concurrent` 且排队超时 | HTTP **503**，message「AI 服务繁忙，请稍后重试」；SSE `event: error` 含 `code: 503` |
| score-image | 输出 token 有上限，整体可能略快、更稳定 |
| 流式分析/拍摄建议 | 用户中途关闭连接，服务端会停止拉取 Kimi 流 |
| 服务重启 | SIGTERM 后最多等待 `shutdown_sec` 再退出 |

---

## 五、部署与调优建议

1. **复制配置**：将 `app.yaml.example` 中新字段合并到本地 `config/app.yaml`（不必改也能用默认值）。
2. **反向代理**：Nginx `proxy_read_timeout` 仍建议 ≥ `kimi.timeout_sec`；SSE 需 `X-Accel-Buffering: no`（代码已设置响应头）。
3. **压测调参**：
   - 单机内存紧张：`max_concurrent` 降到 3，`max_long_edge` 降到 768。
   - Kimi 配额充足、机器性能好：可提高到 8–10，观察 429 率。
4. **未做项**（后续迭代）：结果缓存、429 重试、Prometheus 指标、`recognize-image` 鉴权限流。

## 七、总结

本次改进聚焦 **Kimi 上游治理** 与 **可配置化**，在不改变现有 API 路径和响应格式的前提下，降低高并发雪崩风险、缩短无效上游占用、并让运维可通过 `app.yaml` 调参。核心瓶颈仍在 Kimi 多模态推理时间；本地 Go 预处理与连接管理已进一步优化。
